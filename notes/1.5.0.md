# ModbusLink v1.5.0 版本发布说明

## 🛡️ 核心更新：全协议栈缓冲区清洗 (Universal Buffer Flush)

### 摘要

本次更新专注于提升库在 **高噪声环境** 和 **不稳定网络** 下的健壮性。我们将 v1.4.2 中针对 TCP 引入的缓冲区清洗机制，标准化并扩展到了 **RTU** 和 **ASCII** 协议层。

这意味着无论是通过以太网还是串口（RS485/232），ModbusLink 现在都具备了**自动清除陈旧数据**和**从错序响应中自愈**的能力。

## ✨ 新特性

### 1. 统一的 `flush()` API

在所有 Transport 基类（同步/异步）中增加了标准化的 `flush()` 方法。

- **功能**：非阻塞地读取并丢弃接收缓冲区中的所有剩余数据。
- **返回值**：返回被丢弃的字节数（用于调试或统计噪声）。

```python
# 适用于 TCP, RTU, ASCII 的所有 Transport 对象
discarded = transport.flush()  # Sync
# discarded = await transport.flush()  # Async
```

### 2. RTU/ASCII 的“预防式”自动清洗

串口通信（尤其是 RS485）极易受到电气噪声干扰，且 Modbus RTU 缺乏 TCP 那样的事务 ID (TID) 保护机制。为了解决此问题，v1.5.0 引入了**预防式清洗策略**：

- **变更点**：在 `RTU` 和 `ASCII` 的 `send_and_receive` 方法中，在发送任何请求数据之前，会自动调用 `flush()`。
- **目的**：确保本次请求发出前，缓冲区是绝对干净的。防止上一条指令的迟到响应或线路噪声被误认为是当前指令的响应头。
- **影响**：极大地降低了“CRC 校验错误”和“非法响应”的发生概率。

### 3. TCP 的“响应式”手动清洗

对于 TCP 协议，`flush()` 主要作为异常处理机制存在。

- 无变更：保持 v1.4.2 的逻辑，允许用户在捕获 `TimeOutError` 后手动调用 `flush()` 以复位 TCP 流状态。

## 🛠️ 变更详情

### Transport 层

- **BaseTransport**:
  - 新增抽象方法 `flush()`。
- **RtuTransport / AsciiTransport**:
  - 实现 `flush()`：
    - **同步版**：使用 `serial.reset_input_buffer()`，极速清空。
    - **异步版**：使用 `read(4096)` 循环读取直至超时（0.001s），模拟非阻塞清空。
  - 更新 `send_and_receive`：在 write 操作前强制执行 `flush()`。
- **TcpTransport**:
  - `v1.4.2` 已经实现了 `flush()`，无变更。

## ⚠️ 兼容性说明

- **普通用户**：完全向后兼容，无缝升级。
- **插件开发者**：如果您继承了 BaseTransport 编写自定义传输层，升级后必须实现 flush() 方法，否则实例化时会报错。

## 📈 升级建议

强烈建议所有使用 **串口（RTU/ASCII）** 的用户升级。

此版本解决了在半双工总线或高延迟设备上常见的“请求错位”问题。对于 TCP 用户，无变更。
